Redis项目

  • 项目亮点：

    1. 使用redis来解决集群模式下的session共享问题。

    因为一台tomcat只会存放自己的session。

    • 我们使用redis代替session共享，在登录时会生成token存入redis以校验用户登录状态。
    • 两层拦截器，第一层拦截所有页面，动态刷新token并将用户信息保存到ThreadLocal中。第二层拦截需要用户登录的页面，若ThreadLocal中没有用户则拦截请求。
    • 使用ThreadLocal来做到线程隔离，每个线程保存和操作自己的用户数据。

  1. 使用redis做缓存，并解决缓存穿透、缓存击穿、缓存一致性问题。

    • 我们使用超时剔除的缓存更新策略。
    • 缓存一致性问题：数据更新时，我们先更新数据库，再删除缓存。等下次查询时再将数据缓存到redis更新缓存。
    • 缓存穿透问题：客户端请求的数据在缓存中和数据库中都不存在，这样缓存永远不会生效，这些请求都会打到数据库。

    我们使用缓存空对象的方案解决缓存穿透问题：哪怕这个数据在数据库中也不存在，我们也把这个数据存入到redis中去，这样，下次用户过来访问这个不存在的数据，那么在redis中也能找到这个数据就不会进入到缓存了。

    • 缓存击穿问题：缓存击穿问题也叫热点Key问题，就是一个被高并发访问并且缓存重建业务较复杂的key突然失效了，无数的请求访问会在瞬间给数据库带来巨大的冲击。

    我们使用逻辑过期的策略解决缓存击穿问题，当缓存过期时我们不删除缓存而是设置逻辑过期时间。当一个线程查询缓存发现过期时，获取互斥锁去新开一个线程去查询数据库重建缓存数据，原线程返回旧数据。其他进程来查询发现逻辑过期且获取锁失败则直接返回过期数据。

具体实现：

根据id查询店铺时，如果缓存未命中，则查询数据库，将数据库结果写入缓存，并设置超时时间

根据id修改店铺时，先修改数据库，再删除缓存

③缓存穿透问题

缓存穿透 ：缓存穿透是指客户端请求的数据在缓存中和数据库中都不存在，这样缓存永远不会生效，这些请求都会打到数据库。

常见的解决方案有两种：

  • 缓存空对象
    • 优点：实现简单，维护方便
    • 缺点：
      • 额外的内存消耗
      • 可能造成短期的不一致
  • 布隆过滤
    • 优点：内存占用较少，没有多余key
    • 缺点：
      • 实现复杂
      • 存在误判可能

简单的解决方案就是哪怕这个数据在数据库中也不存在，我们也把这个数据存入到redis中去，这样，下次用户过来访问这个不存在的数据，那么在redis中也能找到这个数据就不会进入到缓存了

④缓存击穿问题

缓存击穿问题也叫热点Key问题，就是一个被高并发访问并且缓存重建业务较复杂的key突然失效了，无数的请求访问会在瞬间给数据库带来巨大的冲击。

常见的解决方案有两种：

  • 互斥锁
  • 逻辑过期

  1. 互斥锁：当未命中缓存时，我们先给进程加锁，让它去查询数据库并写回缓存。与此同时其他进程等待。

  2. 逻辑过期：当缓存过期时我们不删除缓存而是设置逻辑过期时间。当一个线程查询缓存发现过期时，获取互斥锁去新开一个线程去查询数据库重建缓存数据，原线程返回旧数据。其他进程来查询发现逻辑过期且获取锁失败则直接返回过期数据。

⑤缓存血崩问题

缓存雪崩是指在同一时段大量的缓存key同时失效或者Redis服务宕机，导致大量请求到达数据库，带来巨大压力。

解决方案：

  • 给不同的Key的TTL添加随机值
  • 利用Redis集群提高服务的可用性
  • 给缓存业务添加降级限流策略
  • 给业务添加多级缓存

3.1单体系统的超卖问题、一人一单

  • 使用乐观锁解决超卖问题。

乐观锁（Optimistic Locking）是一种并发控制的技术，通常用于数据库操作中，以解决多个用户同时修改同一数据时可能发生的数据冲突问题。在 Java 中，乐观锁通常通过版本号或时间戳来实现。

在乐观锁中，每个数据记录都有一个版本号或时间戳字段。当用户要更新数据时，系统首先会读取数据的当前版本号或时间戳，然后用户在执行更新操作之前会比较这个版本号或时间戳是否发生了变化。如果在用户读取数据后，数据没有被其他用户修改过，则用户的更新操作会成功，同时版本号或时间戳会被更新；但如果在用户读取数据后，数据已经被其他用户修改过，则用户的更新操作会失败，系统会返回错误信息，要求用户重新读取数据并重新尝试更新。

在 Java 中，乐观锁可以通过各种方式实现，例如：

  1. 使用版本号：在数据表中增加一个版本号字段，每次更新数据时增加版本号，当更新数据时检查版本号是否匹配，如果匹配则执行更新操作，否则抛出异常或返回错误信息。
  2. 使用时间戳：在数据表中增加一个时间戳字段，记录数据的最后更新时间，每次更新数据时更新时间戳，当更新数据时比较当前时间戳与之前读取的时间戳是否一致，如果一致则执行更新操作，否则抛出异常或返回错误信息。
  3. 使用乐观锁框架：一些持久化框架（例如 Hibernate）提供了乐观锁的支持，可以通过配置来启用乐观锁机制，框架会自动处理版本号或时间戳的更新和比较。
  • 使用syncronized同步锁来解决一人一单的问题，锁就是ThreadLocal中保存的用户信息。
  • 减库存、创建订单操作需要用@Transactional事务管理。

但是，集群模式下使用syncronized同步锁会失效（因为是不同的tomcat，是不同的线程），因此一人一单的问题不能得到解决，需要用到分布式锁。

3.2使用Redis的SETNX命令实现分布式锁

①问题1：Redis分布式锁存在误删的情况

主要存在的问题：Redis分布式锁误删情况说明

逻辑说明：

持有锁的线程在锁的内部出现了阻塞，导致他的锁自动释放，这时其他线程，线程2来尝试获得锁，就拿到了这把锁，然后线程2在持有锁执行过程中，线程1反应过来，继续执行，而线程1执行过程中，走到了删除锁逻辑，此时就会把本应该属于线程2的锁进行删除，这就是误删别人锁的情况说明

解决方案：解决方案就是在每个线程释放锁的时候，去判断一下当前这把锁是否属于自己，如果属于自己，则不进行锁的删除，假设还是上边的情况，线程1卡顿，锁自动释放，线程2进入到锁的内部执行逻辑，此时线程1反应过来，然后删除锁，但是线程1，一看当前这把锁不是属于自己，于是不进行删除锁逻辑，当线程2走到删除锁逻辑时，如果没有卡过自动释放锁的时间点，则判断当前这把锁是属于自己的，于是删除这把锁。

Redis分布式锁误删问题解决办法：

核心逻辑：在存入锁时，往redis中存入自己线程的标识。在删除锁时，从redis中查询这锁的标识，判断当前这把锁的标识是不是自己存入的。如果是，则进行删除，如果不是，则不进行删除。

3.3分布式系统解决超卖问题——redis分布式锁

①SETNX分布式锁的问题

  • 不可重入

    同一线程无法多次获取同一把锁

  • 不可重试

    获取锁只尝试一次就返回false，没有重试机制

  • 超时释放

    锁超时释放虽然可以避免死锁，但如果是业务执行耗时较长，也会导致锁释放，存在安全隐患

  • 主从一致性

    如果Redis提供了主从集群，主从同步存在延迟，当主宕机时，如果从并同步主中的锁数据，则会出现锁失效。

②使用redisson的分布式锁

redisson 是一个基于 Redis 的分布式 Java 对象服务框架，它提供了一种简单易用的方式来使用 Redis，实现了一系列的分布式 Java 对象服务和功能。Redisson 可以帮助开发者构建高性能、可扩展的分布式系统，它提供了一系列的功能，包括分布式集合、分布式锁、分布式对象、分布式消息队列等等。

Redisson 的主要特点和功能包括：

  1. 分布式对象：Redisson 提供了一系列的分布式对象，如分布式 Map、Set、List、Queue、Deque、Lock、Semaphore、CountDownLatch 等，这些分布式对象可以在分布式环境中共享和操作。
  2. 分布式锁：Redisson 提供了基于 Redis 实现的分布式锁，它能够保证在分布式环境中对共享资源的互斥访问，从而实现分布式系统的并发控制。
  3. 分布式消息队列：Redisson 提供了基于 Redis 实现的分布式消息队列，支持发布订阅模式和点对点模式，能够实现消息的可靠传输和消费。
  4. 异步执行：Redisson 支持异步执行任务，可以将任务提交到 Redisson 的线程池中异步执行，从而提高系统的并发性能。
  5. 事件监听：Redisson 支持事件监听机制，可以监听分布式对象的变化和状态，从而实现分布式系统的实时监控和管理。

总的来说，Redisson 是一个功能丰富、易于使用的分布式 Java 对象服务框架，可以帮助开发者快速构建高性能、可扩展的分布式系统。

5.3关注推送

对于新型的Feed流的的效果：不需要我们用户再去推送信息，而是系统分析用户到底想要什么，然后直接把内容推送给用户，从而使用户能够更加的节约时间，不用主动去寻找。

Feed流产品有两种常见模式：
Timeline：不做内容筛选，简单的按照内容发布时间排序，常用于好友或关注。例如朋友圈

  • 优点：信息全面，不会有缺失。并且实现也相对简单
  • 缺点：信息噪音较多，用户不一定感兴趣，内容获取效率低

智能排序：利用智能算法屏蔽掉违规的、用户不感兴趣的内容。推送用户感兴趣信息来吸引用户

  • 优点：投喂用户感兴趣信息，用户粘度很高，容易沉迷
  • 缺点：如果算法不精准，可能起到反作用

Feed流的三种模式：拉模式、推模式、推拉结合模式

传统了分页在feed流是不适用的，因为我们的数据会随时发生变化

假设在t1 时刻，我们去读取第一页，此时page = 1 ，size = 5 ，那么我们拿到的就是106 这几条记录，假设现在t2时候又发布了一条记录，此时t3 时刻，我们来读取第二页，读取第二页传入的参数是page=2 ，size=5 ，那么此时读取到的第二页实际上是从6 开始，然后是62 ，那么我们就读取到了重复的数据，所以feed流的分页，不能采用原始方案来做。

Feed流的滚动分页

我们需要记录每次操作的最后一条，然后从这个位置开始去读取数据

举个例子：我们从t1时刻开始，拿第一页数据，拿到了10~6，然后记录下当前最后一次拿取的记录，就是6，t2时刻发布了新的记录，此时这个11放到最顶上，但是不会影响我们之前记录的6，此时t3时刻来拿第二页，第二页这个时候拿数据，还是从6后一点的5去拿，就拿到了5-1的记录。我们这个地方可以采用sortedSet来做，可以进行范围查询，并且还可以记录当前获取数据时间戳最小值，就可以实现滚动分页了

核心的意思：就是我们在保存完探店笔记后，获得到当前笔记的粉丝，然后把数据推送到粉丝的redis中去。
